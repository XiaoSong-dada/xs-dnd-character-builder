# 项目开发约定

## 项目目标

本项目是一个移动端优先的 D&D 快速车卡 Web 应用，目标是通过分步引导、自动计算和规则校验，让不了解规则的新玩家也能快速创建并使用合法角色卡。

开始实现前先阅读 [`../docs/rules.md`](../docs/rules.md) 与 [`../docs/frontend-architecture.md`](../docs/frontend-architecture.md)。规则文档定义产品和 D&D 规则边界，前端架构文档定义目录权限、依赖方向和模块定位方式。如果代码行为与文档冲突，以文档为当前项目基线；发现文档有误时，先说明依据并同步修正文档和实现。

## 选择性文档加载

- [`../docs/dnd-classes.md`](../docs/dnd-classes.md) 是职业、子职与职业特性的专项资料，不属于每次任务的立即加载文档。
- 只有任务涉及职业选择、职业等级、子职、职业特性、职业资源、职业前置条件、对应规则数据或 UI 展示时，才读取该文档。
- 任务只涉及基础设施、通用组件、样式、构建、路由或与职业无关的规则时，不得默认加载该文档。
- 已知目标职业时，优先只读取该职业章节及文档中的通用字段约定，不为单职业任务加载无关职业章节。
- `../docs/classes/subclasses/<职业ID>/` 下每个 Markdown 只记录该职业的一个子职，均属于二级选择性加载文档；只有目标职业和子职明确或任务需要该子职规则时，才通过 `dnd-classes.md` 中的链接读取对应文件，禁止一次性加载全部职业目录。
- 修改职业、子职或职业特性的数据结构与行为时，必须同步更新该文档对应章节；未改变职业资料时无需更新。

## 强制技能调用

- 在理解或澄清任何需求、设计实现方案、阅读或修改代码、诊断缺陷、编写测试以及进行代码审查之前，必须先读取并使用项目技能 [`.agents/skills/programming-understanding/SKILL.md`](skills/programming-understanding/SKILL.md)。
- 该技能应自动触发，不需要用户显式写出技能名称。
- 使用技能时，先明确目标产物、作用范围和成功标准，再检查现有文件并形成可验证的实现假设。
- 如果同时存在更具体的领域技能，先读取领域技能，再由 `programming-understanding` 负责需求理解、步骤组织和验证设计。
- 只有纯聊天且不涉及项目需求、代码、规则、文档或实现决策时，才可以不调用该技能。

## 首期范围

- 默认规则集：D&D 5e 2024 修订版。
- 首个可用版本只支持 1 级角色。
- 只采用公开免费规则及许可范围内的 SRD 内容。
- 优先完成标准数组、27 点购点和自定义属性录入。
- 优先支持浏览器本地保存以及 JSON 导入、导出。
- 暂不实现多职业、高等级升级、商业扩展书完整内容和 DM 房规编辑器。
- 不得把 2014 与 2024 规则数据隐式混用。

## 技术约定

- 使用 Vue 3、TypeScript 和 Vite。
- 使用 Vue Router 管理页面入口，使用 Pinia 管理跨页面共享状态。
- 使用 Reka UI 提供无样式交互原语，使用 SCSS 与 CSS Variables 实现项目主题；不引入 Tailwind CSS。
- Vue 组件优先使用 Composition API 与 `<script setup lang="ts">`。
- 开启并保持 TypeScript 严格检查，不使用无原因的 `any`、`@ts-ignore` 或非空断言。
- 业务状态、规则数据、派生计算和 UI 组件应分层组织。
- Vue 组件只负责展示与收集输入，不在组件中硬编码职业、物种、出身或法术规则。
- 派生数值由纯函数计算，不把可计算结果作为唯一事实源重复维护。
- 引入新依赖前先判断是否能用现有能力完成；依赖应有明确用途并适合浏览器环境。
- 所有 `src/` 下的 TypeScript、Vue 和样式模块引用必须使用 `@` 别名，禁止使用 `./` 或 `../` 相对路径。第三方包导入不受此限制。

## 目录权限与依赖收束

- 完整目录权限矩阵、允许依赖和禁止依赖以 [`../docs/frontend-architecture.md`](../docs/frontend-architecture.md) 为准。
- 前端主依赖方向为 `router -> layout/views -> features或页面私有hooks -> stores/services/rules`；不得由下层反向导入 `views`、`layout` 或 `router`。
- `src/router` 只负责路由注册、路由元信息与页面组织，不承载车卡业务逻辑。
- `src/views` 只负责页面级装配；不得直接实现规则计算、持久化或网络请求。
- `src/layout` 只负责布局容器、导航和通用页面骨架，不承载具体车卡步骤。
- `src/features` 按车卡业务能力内聚组件、私有 hooks、配置和装配逻辑，例如属性生成、职业选择和角色卡查看。
- `src/components` 只放确认可跨功能复用的组件；基于 Reka UI 的通用封装统一放 `src/components/ui`。
- `src/hooks` 只放跨多个功能复用的 hooks。只服务一个页面或功能的 hook 必须留在对应 `views/**/hooks` 或 `features/**/hooks`。
- `src/stores` 只管理跨页面共享、需要响应式维护的原始状态，不保存可以由规则函数重新计算的派生值。
- `src/rules` 只放规则集数据、派生计算和合法性校验。该目录必须保持框架无关，不得依赖 Vue、Pinia、Router、DOM、Cookie、存储或网络。
- `src/services` 负责 localStorage、Cookie、JSON 导入导出和未来网络接口等外部边界；页面与通用组件不得直接访问这些外部能力。
- `src/api` 仅在出现真实后端接口后创建，并负责全部请求封装；禁止在 views、components、stores 中直接写请求。
- `src/types` 按业务域统一管理共享类型；只在单个组件内部使用的私有类型可以留在组件目录。
- `src/config` 是环境变量和运行配置的唯一读取入口；其他业务模块不得直接读取 `import.meta.env`。
- `src/constants` 放跨模块稳定常量；单功能常量留在该功能目录，不得为“可能复用”而提前上提。
- `src/styles` 负责全局样式、主题令牌、通用布局工具和必要覆盖；组件私有样式优先保留在组件的 scoped SCSS 中。
- `src/utils` 只放无业务状态、无外部副作用的通用函数，不得成为无法归类代码的收纳目录。

- 单页面或单功能逻辑必须就近内聚；只有存在至少两个明确调用方时，才允许上提为公共 hooks、types、constants、components 或 utils。
- 公共模块不得导入具体业务目录。跨 feature 复用应上提为真正的公共模块，不能直接引用另一个 feature 的内部文件。
- 目录随实际代码演进，不为了匹配结构创建空文件夹。

## 规则数据约定

- 每条规则内容必须明确标记 `ruleset`，例如 `5e-2024`。
- 规则条目使用稳定、可序列化的字符串 ID，不使用展示名称作为关联键。
- 中文名称、英文名称、规则 ID 和来源信息分开保存。
- 熟练、能力、修正、法术和装备效果必须保留来源，界面应能解释数值由来。
- 角色原始选择与派生结果分开：原始选择可以保存，派生结果应能重新计算。
- 条件规则使用可测试的数据结构或规则函数表达，避免散落在模板条件中。
- 非法选择必须返回可供玩家理解的中文原因。

## UI 与移动端约定

- 从手机宽度设计，再扩展到平板和桌面端。
- 主要操作目标保持足够触控面积，不依赖悬停才能使用。
- 分步车卡过程中始终显示当前步骤、完成进度和关键角色摘要。
- 页面刷新或意外关闭后，应能够恢复尚未完成的角色。
- 数值旁应提供来源或计算说明，不能只显示最终结果。
- 错误提示应说明如何修正，不使用只有开发者才能理解的报错。
- 跑团查看模式优先展示 AC、HP、先攻、速度、攻击、技能、法术与可用资源。
- 使用语义化 HTML，并保证表单标签、键盘操作、颜色对比和焦点状态可用。
- 页面日常布局优先复用 `src/styles` 中已有的通用 SCSS 工具；复杂且仅属于单组件的外观留在组件 scoped SCSS 中。
- 不直接修改 Reka UI 内部实现；通用视觉和交互规范通过 `src/components/ui` 二次封装。

## 内容与版权约定

- 不直接复制未开放商业规则书的完整规则正文、职业表、法术或其他受限内容。
- 添加规则数据时记录来源与规则版本，并确认内容处于允许使用的许可范围。
- 对规则的项目内说明优先采用简要转述；需要原文时仅使用许可明确的来源。

## 修改流程

1. 阅读相关规则文档、前端架构文档和现有实现。
2. 通过 `rg` 定位目标入口，检查目标模块的扇入（调用方）与扇出（下游依赖），确认影响的页面、features、hooks、stores、rules、services、types 与公共能力。
3. 明确本次修改影响的规则集、车卡步骤、目录权限和派生数值。
4. 先更新或补充类型与规则测试，再修改界面交互。
5. 用最小范围完成需求，不顺手改动无关代码。
6. 同一修改如果改变规则行为，必须同步更新 `docs/rules.md`。
7. 新增、删除、移动模块或改变依赖方向时，必须同步更新 `docs/frontend-architecture.md` 的当前拓扑。
8. 完成后执行类型检查、测试和生产构建。
9. 涉及 UI 时，在手机尺寸下实际检查完整流程、错误状态与刷新恢复。
10. 交付说明必须写明“已复核前端依赖拓扑，已更新”或“已复核前端依赖拓扑，无需更新”。

## 测试约定

- 属性调整值、熟练加值、HP、AC、技能、攻击和法术 DC 等计算必须有单元测试。
- 规则校验至少覆盖合法输入、边界值、缺少必选项和互斥选择。
- 每个修复过的规则错误都应增加回归测试。
- 组件测试关注用户可见行为，不依赖组件内部实现细节。
- 不以单次构建成功代替规则正确性验证。

## 完成标准

一项功能只有同时满足以下条件才算完成：

- 行为符合 `docs/rules.md` 和当前规则集边界。
- TypeScript 类型检查通过。
- 相关自动化测试通过。
- 生产构建通过。
- 没有引入浏览器控制台错误。
- 涉及界面时已在手机宽度验证主要流程。
- 新玩家能够理解选项、错误原因和数值来源。
- 规则或产品行为发生变化时，相关文档已同步更新。

## Git 约定

- 不提交 `node_modules`、构建产物、编辑器缓存或本地密钥。
- 提交内容保持单一目的，避免混入无关格式化。
- 未经用户明确要求，不提交、不推送、不覆盖用户已有修改。
- 不使用会丢失工作区内容的破坏性 Git 命令。
