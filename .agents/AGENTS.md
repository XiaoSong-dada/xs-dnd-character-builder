# 项目开发约定

## 项目目标

本项目是一个移动端优先的 D&D 快速车卡 Web 应用，目标是通过分步引导、自动计算和规则校验，让不了解规则的新玩家也能快速创建并使用合法角色卡。

开始实现前先阅读 [`docs/rules.md`](docs/rules.md)。如果代码行为与规则文档冲突，以规则文档为当前项目基线；发现规则文档有误时，先说明依据并同步修正文档和实现。

## 强制技能调用

- 在理解或澄清任何需求、设计实现方案、阅读或修改代码、诊断缺陷、编写测试以及进行代码审查之前，必须先读取并使用项目技能 [`.agents/skills/programming-understanding/SKILL.md`](skills/programming-understanding/SKILL.md)。
- 该技能应自动触发，不需要用户显式写出技能名称。
- 使用技能时，先明确目标产物、作用范围和成功标准，再检查现有文件并形成可验证的实现假设。
- 如果同时存在更具体的领域技能，先读取领域技能，再由 `programming-understanding` 负责需求理解、步骤组织和验证设计。
- 只有纯聊天且不涉及项目需求、代码、规则、文档或实现决策时，才可以不调用该技能。

## 首期范围

- 默认规则集：D&D 5e 2024 修订版。
- 首个可用版本只支持 1 级角色。
- 只采用公开免费规则及许可范围内的 SRD 内容。
- 优先完成标准数组、27 点购点和自定义属性录入。
- 优先支持浏览器本地保存以及 JSON 导入、导出。
- 暂不实现多职业、高等级升级、商业扩展书完整内容和 DM 房规编辑器。
- 不得把 2014 与 2024 规则数据隐式混用。

## 技术约定

- 使用 Vue 3、TypeScript 和 Vite。
- Vue 组件优先使用 Composition API 与 `<script setup lang="ts">`。
- 开启并保持 TypeScript 严格检查，不使用无原因的 `any`、`@ts-ignore` 或非空断言。
- 业务状态、规则数据、派生计算和 UI 组件应分层组织。
- Vue 组件只负责展示与收集输入，不在组件中硬编码职业、物种、出身或法术规则。
- 派生数值由纯函数计算，不把可计算结果作为唯一事实源重复维护。
- 引入新依赖前先判断是否能用现有能力完成；依赖应有明确用途并适合浏览器环境。

## 建议目录结构

```text
src/
  components/       通用界面组件
  features/         按车卡步骤划分的业务功能
  rules/            规则数据、规则集注册与校验逻辑
  stores/           角色编辑状态
  types/            共享 TypeScript 类型
  utils/            无业务状态的通用函数
  views/            页面级组件
docs/
  rules.md          规则和产品基线
```

目录应随实际代码演进，不为了匹配结构而创建空文件夹。

## 规则数据约定

- 每条规则内容必须明确标记 `ruleset`，例如 `5e-2024`。
- 规则条目使用稳定、可序列化的字符串 ID，不使用展示名称作为关联键。
- 中文名称、英文名称、规则 ID 和来源信息分开保存。
- 熟练、能力、修正、法术和装备效果必须保留来源，界面应能解释数值由来。
- 角色原始选择与派生结果分开：原始选择可以保存，派生结果应能重新计算。
- 条件规则使用可测试的数据结构或规则函数表达，避免散落在模板条件中。
- 非法选择必须返回可供玩家理解的中文原因。

## UI 与移动端约定

- 从手机宽度设计，再扩展到平板和桌面端。
- 主要操作目标保持足够触控面积，不依赖悬停才能使用。
- 分步车卡过程中始终显示当前步骤、完成进度和关键角色摘要。
- 页面刷新或意外关闭后，应能够恢复尚未完成的角色。
- 数值旁应提供来源或计算说明，不能只显示最终结果。
- 错误提示应说明如何修正，不使用只有开发者才能理解的报错。
- 跑团查看模式优先展示 AC、HP、先攻、速度、攻击、技能、法术与可用资源。
- 使用语义化 HTML，并保证表单标签、键盘操作、颜色对比和焦点状态可用。

## 内容与版权约定

- 不直接复制未开放商业规则书的完整规则正文、职业表、法术或其他受限内容。
- 添加规则数据时记录来源与规则版本，并确认内容处于允许使用的许可范围。
- 对规则的项目内说明优先采用简要转述；需要原文时仅使用许可明确的来源。

## 修改流程

1. 阅读相关规则文档和现有实现。
2. 明确本次修改影响的规则集、车卡步骤和派生数值。
3. 先更新或补充类型与规则测试，再修改界面交互。
4. 用最小范围完成需求，不顺手改动无关代码。
5. 同一修改如果改变规则行为，必须同步更新 `docs/rules.md`。
6. 完成后执行类型检查、测试和生产构建。
7. 涉及 UI 时，在手机尺寸下实际检查完整流程、错误状态与刷新恢复。

## 测试约定

- 属性调整值、熟练加值、HP、AC、技能、攻击和法术 DC 等计算必须有单元测试。
- 规则校验至少覆盖合法输入、边界值、缺少必选项和互斥选择。
- 每个修复过的规则错误都应增加回归测试。
- 组件测试关注用户可见行为，不依赖组件内部实现细节。
- 不以单次构建成功代替规则正确性验证。

## 完成标准

一项功能只有同时满足以下条件才算完成：

- 行为符合 `docs/rules.md` 和当前规则集边界。
- TypeScript 类型检查通过。
- 相关自动化测试通过。
- 生产构建通过。
- 没有引入浏览器控制台错误。
- 涉及界面时已在手机宽度验证主要流程。
- 新玩家能够理解选项、错误原因和数值来源。
- 规则或产品行为发生变化时，相关文档已同步更新。

## Git 约定

- 不提交 `node_modules`、构建产物、编辑器缓存或本地密钥。
- 提交内容保持单一目的，避免混入无关格式化。
- 未经用户明确要求，不提交、不推送、不覆盖用户已有修改。
- 不使用会丢失工作区内容的破坏性 Git 命令。
