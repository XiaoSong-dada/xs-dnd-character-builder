# 编程理解与技能沉淀

这个 Skill 用于先把问题理解清楚，再决定如何修改代码，或者把一套反复出现的处理流程沉淀为可复用的技能文件。

它覆盖两类典型场景：

- 编程任务本身还不够清晰，需要先做目标澄清、代码理解、路径定位、验证设计。
- 当前对话里已经出现一套可复用的方法，需要整理成中文版 `SKILL.md`。

## 何时使用

- 遇到编程问题、代码阅读理解问题、需求理解问题。
- 用户描述含糊，存在多种实现解释，不能直接安全编码。
- 需要先提炼流程，再做最小改动。
- 需要新建、改写、中文化或审查 `SKILL.md`。
- 需要把调试、评审、实现、排查步骤沉淀成可复用规范。

## 不适用场景

- 任务已经非常明确，只需要做一个小改动且不需要额外流程沉淀。
- 问题属于强领域专用技能，且现有专用 Skill 已经足够覆盖，例如 BMF 宏命令细节；这类场景应优先叠加专用 Skill。

## 工作目标

使用本 Skill 时，默认目标是：

- 明确要产出的结果是什么。
- 把问题缩小到一个可验证的局部切片。
- 形成一个可证伪的理解或实现假设。
- 给出最小可行改动、最小可行解释，或最小可行技能草案。
- 明确完成检查与后续迭代点。

## 工作流程

### 1. 先定结果，再决定动作

先判断本次任务要交付的主结果：

- 代码修改。
- 问题解释。
- 技能文件。
- 规范补充。

至少补齐以下信息：

- 目标对象：代码、文档、技能、说明。
- 作用范围：单文件、单流程、工作区级、个人级。
- 成功标准：改好什么、解释清什么、沉淀出什么。

如果这些信息缺失到无法安全推进，就先提出最少量的澄清问题。

### 2. 抽取当前对话里的真实流程

优先从当前对话、当前文件、附近实现里抽取已经出现的方法，而不是凭空设计流程。

要提炼的内容包括：

- 正在遵循的步骤顺序。
- 每一步的判断条件。
- 哪些情况下需要回退、追问或切换路径。
- 完成后如何验证结果。

如果已经存在稳定模式，就复用现有模式；不要为一次性场景发明复杂框架。

### 3. 判断是否需要澄清

当没有形成清晰流程时，优先澄清这几类问题：

- 目标产物是什么。
- 技能是工作区共享还是个人使用。
- 需要快速清单还是完整多步骤工作流。
- 是让 Skill 自动发现，还是只作为斜杠命令手动触发。

如果已经能够形成单一、可执行的工作流，就不要为了“更完整”继续追问。

### 4. 小步起草

如果任务是编程或理解问题：

1. 从最具体的锚点开始，例如当前文件、符号、报错、调用点。
2. 只读取足够多的上下文，形成一个可证伪的局部假设。
3. 设计一个最便宜的判别检查。
4. 在确认切片后，再做最小改动或最小解释。

如果任务是创建 Skill：

1. 先选作用域，项目共享优先放到 `.github/skills/<name>/SKILL.md`。
2. 技能目录名与 `name` 字段保持一致，使用小写英文和连字符。
3. `description` 要写清“做什么”和“何时使用”，并包含触发关键词。
4. 正文至少写清用途、适用时机、步骤、判断规则、完成标准。
5. 若任务要求长期默认生效，再补充 `copilot-instructions.md` 这类 instructions，而不是把“始终生效”错误塞进 Skill 本身。

### 5. 验证

创建或修改完成后，至少检查：

- 技能路径是否正确。
- 文件夹名是否与 `name` 一致。
- YAML frontmatter 是否合法。
- `description` 是否足够具体，能让模型发现该 Skill。
- 如果要求“优先使用”，是否已经把该规则写入工作区 instructions。

### 6. 识别薄弱点并迭代

完成初稿后，主动找最模糊的部分，例如：

- 技能触发范围太宽或太窄。
- 步骤只讲原则，没有讲动作。
- 适用场景和不适用场景边界不清。
- 缺少完成检查。

若这些问题会明显影响使用效果，就继续追问或继续收紧表述；否则保持最小可用版本，不做过度设计。

## 决策规则

- 任务要求“多数会话都生效”的常驻约束：放 instructions，不放 Skill。
- 任务要求“按需触发”的多步骤流程：放 Skill。
- 任务只是一次性、单目标动作：直接修改，不必强行沉淀 Skill。
- 如果已有领域专用 Skill，优先叠加领域 Skill，再用本 Skill 负责问题理解和流程组织。

## 完成标准

只有在以下条件满足时，才算使用本 Skill 完成任务：

- 已明确本次任务的目标结果与作用范围。
- 已形成可执行的步骤，而不是只有抽象原则。
- 已给出至少一个可验证的完成检查。
- 若创建的是 Skill，文件位置、命名和描述均可被正确发现。
- 若用户要求“优先调用”，相应规范已落到工作区级 instructions。

## 输出要求

使用本 Skill 处理任务时，默认应输出：

- 这次任务的目标结果。
- 当前采用的步骤或工作流。
- 是否存在需要澄清的关键分歧。
- 最小草案、最小改动或最小解释。
- 验证方式。
- 如有必要，下一步适合继续沉淀的相关技能或规范。
